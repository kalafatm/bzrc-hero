Create SMSA SOAP client helper

4
Build SMSA payload mapper
build_smsa_payload(shipment, items) — maps our DB fields to SMSA addShipPDF parameters. City name string (not code), ISO2 country, codAmt, shipper details.
server/main.py
5
Abstract submit endpoint (carrier router)
Modify POST /shipments/{id}/submit: check shipment.carrier_code, route to Naqel or SMSA. Extract current Naqel logic into _submit_naqel(), add _submit_smsa().
server/main.py
6
Parse SMSA response + store AWB/label
addShipPDF returns AWB string + base64 PDF. Parse XML response, store in same DB fields as Naqel.
server/main.py

Phase 2: Tracking + Rate Calculation Track

7
SMSA tracking: parse getTrack response
parse_smsa_tracking(xml_result) — extract TrackDetails[] from XML DataSet response. Map Activity/Details/Location/evtDate to our tracking_events schema.
server/main.py
8
Abstract track endpoint (carrier router)
Modify POST /shipments/{id}/track: route to GN Connect or SMSA getTrack. Events stored in same tracking_events table.
server/main.py
9
SMSA incremental polling (getShipUpdates)
Background poller: store smsa_last_row_id as cursor. Poll new updates since last row. More efficient than per-AWB polling.
server/main.py

9. No DB Schema Changes Needed

Good news: The existing shipments, shipment_items, and tracking_events tables already support SMSA. The carrier_code field differentiates carriers. AWB, label, payloads, tracking events all fit the existing schema.

10. Environment Variables (New)

SMSA uses HTTP (not HTTPS) for their SOAP endpoint. This is their official URL. The passKey parameter in every call provides authentication.
